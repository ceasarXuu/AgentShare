# 快速验证工作流

适用于需要快速判断一个产品想法是否值得深入研究的场景。

## 目标

在 1-3 天内完成:
- 痛点存在性验证
- 市场规模粗估
- 竞品快速扫描
- Go/No-Go 决策

## 前置准备

1. 明确要验证的产品想法
2. 设置环境变量 (BRAVE_API_KEY 必需)
3. 创建输出目录: `mkdir -p research_output/quick_validation`

---

## 前置检查: 三道防火墙 (新增)

在开始验证前,先执行三道防火墙检查:

使用 `agents/NBResearcher/SKILLS/three-firewall-check/SKILL.md`

1. **致命风险检查**: 涉及人身安全/财产损失/法律风险? → 任一项是则终止
2. **基本可行性检查**: 6个月内<50万启动? 3个月内MVP? → 2个以上否则终止
3. **个人适配性检查**: 真心想做? 能承受失败? → 少于4个是则慎重

**全部通过后才进入 Day 1**

---

## Day 1: 痛点快速验证

### 步骤 1: 定义核心假设
明确要验证的核心假设:
- 用户痛点: [具体描述]
- 目标用户: [用户画像]
- 解决方案: [产品概念]

### 步骤 2: 快速信源检索
使用 `SKILLS/search-strategy/SKILL.md` 的快速模式:

**搜索查询模板**:
```
1. "[痛点关键词] reddit"
2. "[痛点关键词] complaints 2025"
3. "alternative to [现有方案]"
4. "willing to pay for [功能]"
```

工具使用:
- Brave Search: 执行 4-6 个关键搜索
- Playwright: 抓取 Reddit 热门讨论 (Top 3 threads)

### 步骤 3: 三角验证
寻找 3 个独立证据:
- [ ] Reddit 讨论 (至少 1 个高赞帖子)
- [ ] X.com 抱怨 (至少 10 条相关推文)
- [ ] App Store 差评 (至少 5 条相关评论)

### 步骤 4: 痛点评分
| 维度 | 评分 (1-5) | 证据 |
|------|-----------|------|
| 频率 (多少人遇到) | ... | ... |
| 强度 (痛苦程度) | ... | ... |
| 付费意愿 | ... | ... |

**总分 < 9**: 考虑 No-Go
**总分 9-12**: 需要更多验证
**总分 > 12**: 继续 Day 2

---

## Day 2: 市场与竞品扫描

### 步骤 1: 市场规模粗估
快速估算 TAM:
```
TAM = 目标用户数 × 年付费金额
例: 100万 iPhone 用户 × $30/年 = $3000万
```

判断标准:
- TAM < $1000万: 可能太小
- TAM $1000万-$1亿: 小而美
- TAM > $1亿: 大市场

### 步骤 2: 竞品快速识别
工具使用:
- Brave Search: "[产品类别] app ios"
- GitHub Search: 查找开源替代品
- App Store: 搜索相关关键词

目标: 识别 Top 3-5 竞品

### 步骤 3: 竞品快速分析
对每个竞品记录:
| 竞品 | 下载量估算 | 定价 | 评分 | 主要抱怨 |
|------|-----------|------|------|---------|
| A    | ...       | ...  | ...  | ...     |

### 步骤 4: Gap 识别
快速回答:
- [ ] 竞品是否解决了核心痛点?
- [ ] 用户对现有方案满意吗?
- [ ] 是否存在明显的功能或体验 Gap?

**如果 3 个都是 No**: 考虑 No-Go
**如果至少 2 个是 Yes**: 继续 Day 3

---

## Day 3: 可行性与决策

### 步骤 1: 技术可行性快速评估
回答 3 个问题:
1. 是否需要私有 API? (Yes = 高风险)
2. 是否需要复杂后端? (Yes = 高成本)
3. 是否有技术参考? (No = 高难度)

工具使用:
- GitHub Search: 查找类似实现

### 步骤 2: 执行能力评估
| 维度 | 需要 | 现有 | Gap |
|------|------|------|-----|
| 开发时间 | ... | ... | ... |
| 技术能力 | ... | ... | ... |
| 资金预算 | ... | ... | ... |

### 步骤 3: 风险快速扫描
使用 `SKILLS/risk-assessment/SKILL.md` 的快速模式:
- [ ] App Store 政策风险 (会被拒吗?)
- [ ] 竞争风险 (巨头会进入吗?)
- [ ] 市场风险 (需求会消失吗?)

### 步骤 4: Go/No-Go 决策

**Go 的条件** (必须全部满足):
- [x] 痛点评分 > 9
- [x] 存在明显的竞品 Gap
- [x] 技术可行且在能力范围内
- [x] 无致命风险
- [x] TAM > $1000万

**No-Go 的信号** (任一满足):
- [ ] 痛点评分 < 9
- [ ] 竞品已完美解决
- [ ] 技术不可行或超出能力
- [ ] 存在致命风险 (如政策)
- [ ] 市场太小

- TAM 估算: $XXX
- 目标用户: XXX

## 竞品扫描
| 竞品 | 优势 | 劣势 | Gap |
|------|------|------|-----|
| ...  | ...  | ...  | ... |

## 时间分配建议

- Day 1 上午: 定义假设 + 信源检索 (3-4 小时)
- Day 1 下午: 三角验证 + 痛点评分 (3-4 小时)
- Day 2 上午: 市场规模 + 竞品识别 (3-4 小时)
- Day 2 下午: 竞品分析 + Gap 识别 (3-4 小时)
- Day 3 上午: 可行性评估 (2-3 小时)
- Day 3 下午: 决策 + 报告撰写 (2-3 小时)

**总计: 16-22 小时**

---

## 常见陷阱

1. **确认偏误**: 只寻找支持想法的证据
   - 缓解: 主动搜索 "[idea] doesn't work"

2. **过度乐观**: 高估市场规模
   - 缓解: 使用保守估算

3. **忽略竞品**: 认为没有竞品是好事
   - 缓解: 无竞品可能意味着无需求

4. **技术盲目自信**: 低估开发难度
   - 缓解: 查找类似项目的开发时间
