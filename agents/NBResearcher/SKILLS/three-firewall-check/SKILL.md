# Three Firewall Check - 三道防火墙

## 技能概述

这是"好生意评估"的第一道关卡。任何商业机会在进入详细评估前,必须先通过这三道防火墙。**任何一道防火墙不通过,立即终止评估**。

## 核心原则

> **预防大于治疗**: 用5分钟避免浪费5个月

三道防火墙的设计目的是:
1. 识别致命风险 (防火墙1)
2. 验证基本可行性 (防火墙2)
3. 确认个人适配性 (防火墙3)

## 防火墙1: 致命风险检查 (一票否决)

### 目的
识别可能导致严重后果的致命风险,保护开发者避免承担不可承受的责任。

### 检查清单

```
□ 会不会导致人身伤害或生命危险?
   示例: 医疗建议、安全监控、驾驶辅助
   
□ 会不会导致重大财产损失 (用户的)?
   示例: 金融投资、保险理赔、法律建议
   
□ 失败会不会有严重法律后果?
   示例: 涉及隐私泄露、儿童保护、金融监管
   
□ 需要承诺可靠性但无法保证?
   示例: 紧急求助、关键数据备份、安全防护
   
□ 涉及敏感群体 (老人/儿童/病患) 的安全责任?
   示例: 老人监护、儿童定位、病情监测
```

### 判断标准

- ✅ **通过**: 全部为"否"
- ❌ **不通过**: 任何一个为"是"

### 不通过时的处理

如果防火墙1不通过:
1. **优先选项**: 立即放弃该项目
2. **备选方案**: 大幅调整产品定位,规避致命风险
   - 例如: 从"医疗诊断"改为"健康记录管理"
   - 例如: 从"投资建议"改为"投资数据展示"

**重要**: 不要试图通过"免责声明"来规避致命风险,法律保护力度有限。

### 典型案例

#### ❌ 案例 1: 无恙App
- **描述**: 老人紧急求助工具,强力通知确保儿女收到
- **致命风险**: 
  - ✓ 涉及老人安全责任
  - ✓ 需要承诺可靠性(100%送达)但无法保证
  - ✓ 失败可能导致人身伤害
- **结论**: 不通过防火墙1

#### ✅ 案例 2: 老人关怀提醒App
- **描述**: 提醒儿女关心父母,记录关怀历史
- **致命风险**: 无
- **结论**: 通过防火墙1

## 防火墙2: 基本可行性检查

### 目的
验证项目在资源、时间、技术上的基本可行性,避免启动不切实际的项目。

### 检查清单

```
□ 能否在6个月内用 <50万 启动?
   
□ 能否在3个月内做出可测试的MVP?
   
□ 是否有明确的首批目标用户 (能联系到)?
   
□ 是否理解技术实现路径 (不需要重大技术突破)?
   
□ 是否清楚前3个月的行动计划?
```

### 判断标准

- ✅ **通过**: 全部为"是"
- ⚠️ **需要调整**: 有1-2个"否"
- ❌ **不通过**: 超过2个"否"

### 不通过时的处理

如果有超过2个"否":
1. **缩小范围**: 降低MVP的范围,减少启动成本
2. **分阶段**: 将大项目拆分为多个小阶段
3. **补充准备**: 先解决关键不确定性(如技术验证、用户访谈)

### 详细说明

#### ① 启动资金: <50万,6个月内

**为什么是50万?**
- 对个人开发者来说,这是一个可承受的风险上限
- 超过这个数字,失败的代价太大

**包含什么?**
- 开发成本(人力或外包)
- 服务器和第三方服务
- 设计和其他必需开支
- 6个月的生活成本(如全职)

**如果超过怎么办?**
- 缩小MVP范围
- 延长开发周期,兼职开发
- 寻找合伙人分摊成本

#### ② MVP时间: <3个月

**为什么是3个月?**
- 超过3个月,市场可能变化
- 热情容易消退
- 机会成本太高

**MVP的定义**:
- 最小可测试版本
- 能验证核心假设
- 可以给真实用户使用

**如果超过怎么办?**
- 重新定义MVP,只保留最核心功能
- 使用低代码/无代码工具
- 寻找现有解决方案组合

#### ③ 首批用户: 明确且可联系

**为什么重要?**
- 验证需求的唯一方法
- 获得早期反馈
- 冷启动的关键

**"明确且可联系"是指**:
- 知道他们在哪(论坛、社群、线下)
- 可以直接接触(不是通过广告)
- 愿意尝试新产品

**如果没有怎么办?**
- 先做用户研究,找到目标用户群
- 加入相关社群,建立联系
- 重新思考产品定位

#### ④ 技术实现: 清晰路径

**"不需要重大技术突破"是指**:
- 使用成熟的技术栈
- 参考已有类似产品
- 主要问题可以通过搜索解决

**如果不确定怎么办?**
- 先做技术原型,验证关键难点
- 咨询有经验的开发者
- 降低技术复杂度

#### ⑤ 行动计划: 前3个月清晰

**为什么是前3个月?**
- 足够覆盖MVP开发
- 不会过度计划
- 保持灵活性

**应该包含什么?**
- 每个月的主要里程碑
- 需要做的关键决策点
- 资源需求和时间分配

## 防火墙3: 个人适配性检查

### 目的
确认这个项目适合你,避免因动机不足或资源不匹配而半途而废。

### 检查清单

```
□ 我是否真的想做这件事 (不是为了赚快钱)?
   
□ 我是否有相关领域经验或强烈兴趣?
   
□ 我能否接受前6个月几乎没有收入?
   
□ 我能否承受失败 (时间和资金损失)?
   
□ 我的家人是否支持?
```

### 判断标准

- ✅ **通过**: 至少4个"是"
- ⚠️ **需要慎重**: 3个"是"
- ❌ **不建议**: 少于3个"是"

### 不通过时的处理

如果少于4个"是":
1. **等待时机**: 先积累资源和经验
2. **降低投入**: 以副业形式开始,不要全职
3. **寻找合伙人**: 补充你缺失的资源或动力

### 详细说明

#### ① 真实动机

**为什么重要?**
- 创业很苦,没有热情坚持不下去
- "赚快钱"心态容易导致错误决策

**如何判断?**
- 如果这个项目不赚钱,你还愿意做吗?
- 你是否愿意在这个领域深耕3-5年?
- 你是否真正关心用户的问题?

#### ② 领域经验/兴趣

**为什么重要?**
- 有经验更容易理解用户需求
- 有兴趣才有动力持续学习
- 降低决策失误的概率

**"相关经验"包括**:
- 工作经验
- 重度用户经验
- 相关行业观察

**如果没有怎么办?**
- 先深入研究1-2个月
- 与该领域的人深度交流
- 作为用户深度体验现有产品

#### ③ 财务准备

**为什么是6个月?**
- 大多数产品需要6个月以上才有收入
- 保守估计,实际可能更长

**包括什么?**
- 生活成本
- 项目开发成本
- 紧急备用金

**如果没有准备好怎么办?**
- 兼职开发,不要辞职
- 降低生活成本
- 寻找天使投资或合伙人

#### ④ 风险承受力

**需要承受的风险**:
- 6-12个月时间"浪费"
- 启动资金损失
- 机会成本(放弃的工作机会)

**如何判断?**
- 即使失败,是否可以回到原来的轨道?
- 失败后是否会严重影响生活?
- 是否有退路和备选方案?

#### ⑤ 家人支持

**为什么重要?**
- 创业会占用大量时间和精力
- 短期内可能影响家庭收入
- 情绪压力需要家人理解

**如何获得支持?**
- 诚实沟通计划和风险
- 明确时间限制(如1年,不成功就回去上班)
- 让家人参与决策

## 执行流程

### Step 1: 准备项目信息

收集以下信息:
- 产品概述(一句话描述)
- 目标用户是谁
- 解决什么问题
- 大致的实现方式
- 初步的成本估算

### Step 2: 依次执行三道防火墙

```
开始
│
├─ 执行防火墙1: 致命风险检查
│   ├─ 不通过 → ❌ 终止评估,输出原因
│   └─ 通过 → 继续
│
├─ 执行防火墙2: 基本可行性检查  
│   ├─ 不通过 → ❌ 终止评估,输出原因和调整建议
│   └─ 通过 → 继续
│
├─ 执行防火墙3: 个人适配性检查
│   ├─ 不通过 → ⚠️ 给出警告,建议等待时机
│   └─ 通过 → ✅ 全部通过,可以进入详细评估
```

### Step 3: 输出结果

**格式**:

```markdown
# 三道防火墙检查结果

## 项目: [项目名称]

### 防火墙1: 致命风险检查
- 状态: ✅通过 / ❌不通过
- 风险清单: [如有]
- 建议: [如需要]

### 防火墙2: 基本可行性检查  
- 状态: ✅通过 / ⚠️需要调整 / ❌不通过
- 问题项: [哪些为"否"]
- 建议: [调整方案]

### 防火墙3: 个人适配性检查
- 状态: ✅通过 / ⚠️慎重考虑 / ❌不建议
- 是的数量: _/5
- 建议: [如需要]

### 最终结论
[✅ 全部通过,可以进入详细评估 / ❌ 不建议继续 / ⚠️ 调整后重新评估]

### 下一步行动
1. ...
2. ...
```

## 使用场景

### 场景1: 快速筛选多个 idea

当有多个产品 idea 需要选择时:
1. 对每个 idea 执行三道防火墙
2. 只对全部通过的 idea 进行详细评估
3. 节省时间,聚焦最可行的方向

### 场景2: 项目早期决策

在投入大量资源前:
1. 用三道防火墙快速验证基本假设
2. 识别致命问题
3. 避免"沉没成本谬误"

### 场景3: 团队内部对齐

当团队对是否做某个项目有分歧时:
1. 用三道防火墙作为客观判断标准
2. 聚焦讨论具体的检查项
3. 基于事实做决策,而非感觉

## 常见问题

### Q1: 有一个检查项不通过,但我觉得可以解决,能继续吗?

A: 关键看是哪一道防火墙:
- **防火墙1**: 绝对不能继续,除非完全改变产品定位
- **防火墙2**: 如果有明确的解决方案,可以先解决再评估
- **防火墙3**: 可以降低投入程度(如兼职做),但要认清风险

### Q2: 防火墙2有1个"否",但其他都是"是",需要停止吗?

A: 不一定停止,但要先解决那个"否":
- 如果是"没有首批用户" → 先做用户研究
- 如果是"技术路径不清晰" → 先做技术原型
- 如果是"启动成本过高" → 缩小MVP范围

### Q3: 我很想做这个项目,但防火墙3只有2个"是",怎么办?

A: 这是最危险的情况,强烈建议:
1. 诚实面对自己的动机和资源
2. 等待时机,先积累
3. 或寻找合伙人补充你缺失的部分

不要因为"很想做"而忽视风险信号。
