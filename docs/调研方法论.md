## 核心哲学

**三大原则**
1. **只相信行为,不相信观点** - 搜索量、付费行为、真实吐槽 > 媒体报道、专家预测
2. **三角验证法则** - 任何结论必须有3个独立来源支撑
3. **自下而上建模** - 从具体可执行路径推导,而非从宏大市场倒推

## 第二阶段:检索技巧与查询优化 (Day 2-4)

### 2.1 四阶段搜索策略

#### Stage 1: 广泛探索 (Broad Discovery)
**目标:** 发现未知的unknown unknowns

```
查询模式:
• "iPhone users complaints 2025"
• "iOS frustration" site:reddit.com
• "wish iPhone had" site:reddit.com
• "[job role] + pain + app"

反模式(严禁):
✗ "best iOS app ideas 2024" → 媒体排行榜
✗ "most profitable apps" → 过时+PR
```

#### Stage 2: 痛点深挖 (Pain Point Validation)
**目标:** 验证痛点的广度和深度

```
查询模式:
• "iPhone battery 96% is this normal" (用户真实语言)
• "iOS 18 bugs reddit" (特定版本)
• "[竞品名] doesn't work" (解决方案gap)

技巧:
✓ 使用长尾关键词
✓ 搜索具体数值("60GB system data")
✓ 避免技术术语(用"why so slow"不用"performance optimization")
```

#### Stage 3: 负面情绪检索法
**目标:** 用户愤怒时最诚实

```
Search Query模板:
• site:reddit.com "I hate" + [industry/task]
• site:reddit.com "alternative to" + [Top Competitor]
• "[app name] + too complicated + reddit"
• "willing to pay for" + [feature]

案例:
搜索"nursing app too complicated"
→ 发现护士讨厌复杂排班工具
→ 产品机会: 简单专注的垂直工具
```

#### Stage 4: Excel替代检索法
**逻辑:** Excel是所有工具类App的最大竞争对手

```
Search Query:
• "[Job Role] + spreadsheet template + free"
• "Excel template for [task]"

洞察:
如果有人专门做Excel模板管理排班/库存
→ 现有App没解决好或太贵
→ 将Excel逻辑App化 = 好产品
```

### 2.2 高级搜索运算符

```
Site限定:
site:reddit.com [keyword]
site:x.com [keyword]
site:producthunt.com [keyword]

时间限定:
搜索后选择"过去1年"

排除干扰:

### 2.3 查询优化实战案例

**案例:探索"电池健康焦虑"机会**

```

## 第三阶段:信息验证与交叉比对 (Day 4-6)

### 3.1 三角验证法 (Triangulation)

**规则:** 任何重要结论必须3个独立来源支撑

**案例:验证"iOS系统数据异常"痛点**

```
来源1: Reddit
r/iPhone: "System Data 62GB"
• 帖子: 50+
• Upvotes: 平均500+
• 共识: "我也是!"

来源2: X.com
搜索: "iOS system data bug"
• 推文: 200+
• 时间跨度: 2024.6至今
• 情绪: 愤怒、困惑

来源3: Apple Community
"iOS 18 Storage Issues"
• 回复: 2000+
• 官方: "请重启"(无实质解决)
• 持续: 6个月+未解决

验证结果: ✅✅✅ 真实、广泛、持续的痛点
```

### 3.2 信号vs噪声识别

**真实信号(Signal)特征:**
```
✅ 多个独立用户相似frustration
✅ 问题持续存在(非一次性bug)
✅ 用户willing to pay或已自行解决
✅ 有清晰context(设备型号、版本)
```

**噪声(Noise)特征:**
```
❌ 仅1-2个用户抱怨
❌ 过于模糊("iPhone太慢")
❌ 极端案例(单个硬件故障)
❌ 已有完美免费方案
```

**对比案例:**

```
Signal示例:
"Battery dropped 100%→96% in 3 months, normal?"
Upvotes: 850 | Comments: 200+
✅ 具体数据 ✅ 大量共鸣 ✅ 反复出现
→ 真实信号

Noise示例:
"My iPhone exploded"
Upvotes: 12 | Comments: "Fake"
❌ 极端个例 ❌ 社区不信 ❌ 无法复现
→ 噪声,忽略
```

### 3.3 数据时效性验证

**规则:仅使用12个月内数据**

**为什么?**
```
• iOS每年major update(17→18→19)
• 用户行为快速变化(AI应用崛起)
• 竞争格局演变(新app崛起、老app衰落)
```

**检查Checklist:**
```
□ 发布日期?
  2024-2025 ✅
  2022-2023 ⚠️ 需谨慎
  2021之前 ❌ 直接忽略

□ 有更新版本?
  Sensor Tower Q4 2024 ✅
  App Annie 2023年报 ❌ 找Q1 2025

□ Major事件影响?
  "2019年screen time数据"
  → 2020疫情改变一切
  → 过时
```

**特例:永恒真理**
```
某些原则不受时间影响:
• 用户厌恶复杂操作
• Freemium转化率2-5%
• 电池焦虑是人性
→ 可参考older research
```

### 5.5 Unit Economics(单位经济模型)

**关键指标定义:**

```
CAC: Customer Acquisition Cost(获客成本)
LTV: Lifetime Value(客户终身价值)
Payback Period: 回本周期
Churn Rate: 流失率
```

**案例计算:**

```
Year 1假设:

CAC计算:
营销支出: $20K
获得用户: 2,500
CAC = $20K / 2.5K = $8/用户

LTV计算:
ARPU: $35/年
平均留存: 2年
Gross Margin: 70%(扣Apple抽成)
LTV = $35 × 2 × 70% = $49

LTV/CAC Ratio:
= $49 / $8 = 6.1x
✅ 健康指标 (>3x为good)

敏感度分析:
如果Churn +20%:
留存: 2年 → 1.6年
LTV: $49 → $39
LTV/CAC: 6.1x → 4.9x
结论: 仍健康,但需注意留存
```

### 6.1 风险识别4维度

**技术风险:**
```

• 依赖私有API? (高风险)
• iOS版本兼容性? (中风险)
• 技术可行性验证? (未验证=高)
```

**市场风险:**
```
评估标准:
• 痛点validated? (未验证=高)
• 竞争激烈程度? (红海=高)
• 市场增长/萎缩? (萎缩=高)
```

**监管风险:**
```
评估标准:
• App Store政策合规? (不合规=致命)
• 隐私政策GDPR? (不合规=致命)
• Apple可能封杀? (高可能=高)
```

**执行风险:**
```
评估标准:
• 团队能力匹配? (不匹配=高)
• 资金runway? (<6月=高)
• 时间预估realistic? (过度乐观=中)
```

### 6.2 风险-影响矩阵

```
      高影响
        ↑
        |
  ④ 灾难 | ③ 严重
  ───────┼─────── 高概率
  ② 注意 | ① 接受
        |
        ↓
      低影响

案例:设备健康助手

① 接受区:
• App Store审核延迟1周
  影响: 轻微
  应对: 提前submit

② 注意区:
• 服务器短暂宕机
  影响: 用户体验下降
  应对: 备份方案

③ 严重区:
• Apple推出类似原生功能
  影响: 核心价值削弱
  应对: 快速pivot到社区功能

④ 灾难区(必须避免):
• 承诺技术上不可能的功能
  影响: 差评如潮,被下架
  应对: 绝不进入这个区!
```

## 第七阶段:常见陷阱与规避 (持续)

### 7.1 调研陷阱

**陷阱1: Confirmation Bias(确认偏误)**
```
表现:
你想做AI照片整理
→ 只搜"AI photo"正面信息
→ 忽略"用户不在乎整理"信号
→ 错误结论"市场需求大"

规避:
✅ 主动寻找反例
✅ 搜索"[idea] doesn't work"
✅ 问"为什么用户不需要?"
```

**陷阱2: Survivorship Bias(幸存者偏差)**
```
表现:
分析成功app: "都有gamification!"
→ 结论: "我也要!"
→ 忽略: 1000个失败app也有

规避:
✅ 同时研究成功和失败案例
✅ 寻找causal关系,不是correlation
✅ 问"没有这个也成功的案例?"
```

**陷阱3: 过度依赖单一数据源**
```
表现:
1篇Sensor Tower: "VPN收入高"
→ 立刻决定做VPN
→ 忽略: 被巨头垄断

规避:
✅ 三角验证(3+来源)
✅ 区分"市场大"和"机会多"
✅ 深挖数据背后context
```

### 7.2 分析陷阱

**陷阱4: Feature-Focused vs Problem-Focused**
```
错误思维:
"我们要做AI+blockchain+AR应用!"
→ 技术驱动,无用户需求

正确思维:
"用户痛点: 电池焦虑
→ 解决方案: 社区对比+心理干预
→ 技术: 简单数据展示就够"
```

**陷阱5: TAM陷阱**
```
错误:
"10亿iPhone用户,拿0.1%也有100万!"
→ 无实际path,空中楼阁

正确:
"Day 1: ASO优化,预计10个organic下载
Week 1: 转化5%,有0.5个付费
→ 需优化到50下载/天才能达标"
```

**陷阱6: "We have no competition"**
```
错误:
"没有exactly相同产品,无竞争!"
→ 忽略替代品和现有方案

正确:
"直接竞品少,但用户用iOS原生+Google
→ 我们在跟'免费+勉强够用'竞争
→ 必须10x better才能让人换"
```

### 7.3 执行陷阱

**陷阱7: 过度设计**
```
表现:
"先做iOS,Android,Web,macOS!"
→ 6个月后还没上架
→ 竞品已占领市场

规避:
✅ MVP思维: 最小可行产品
✅ 单一平台(iOS)先验证
✅ 1个核心功能做到极致
```

**陷阱8: 忽视ASO**
```
错误:
"产品好自然有人用"
→ 无人发现你的应用
→ 下载量0

正确:
✅ 上架前准备ASO策略
✅ 关键词研究
✅ Icon A/B testing
✅ 前10个评价至关重要
```

**陷阱9: Premature Scaling**
```
表现:
下载100,转化2%
→ "投$10K广告快速增长!"
→ 烧钱后发现留存差
→ 亏损

规避:
✅ 先优化转化和留存
✅ Unit economics健康后再scale
✅ LTV/CAC>3才考虑付费获客
```

### 7.4 心态陷阱

**陷阱10: Sunk Cost Fallacy**
```
表现:
"已开发3个月,必须继续!"
→ 即使市场验证失败
→ 继续投入
→ 更大损失

规避:
✅ 设定clear的pivot/kill指标
✅ "如果Week 4下载<100,pivot"
✅ 过去投入是sunk,看未来return
```

**陷阱11: 害怕competition**
```
表现:
"有人做类似产品,我不做了"
→ 放弃validated需求

规避:
✅ 竞争=市场存在证明
✅ 关键是differentiation
✅ 即使Uber成功,Lyft也成功
```

## 最后的黄金法则

### 七大铁律

**1. 数据>直觉,但数据需要解读**
```
看到"VPN收入高"
≠ "你应该做VPN"

要问:
• 为什么高?
• 我能复制吗?
• 壁垒在哪?
```

**2. Validated需求>酷炫idea**
```
"AI+blockchain+AR照片助手"
<
"帮用户缓解电池焦虑"
```

**3. 10个浅分析 < 3个深分析**
```
不要每个方向只花30分钟
选3个深挖3小时
```

**4. 竞争是好事**
```
有竞品 = 有市场
无竞品 = 可能没需求
关键是differentiation
```

**5. 快速失败优于慢速成功**
```
2周发现不行 → 损失2周
6个月后发现不行 → 损失6月+$50K
```

**6. 用户说的 ≠ 用户要的**
```
用户说: "要更快的马"
Ford造了: 汽车

用户说: "电池优化"
你做: "焦虑缓解"
```

**7. 永远保持怀疑**
```
• 这个数据可能错吗?
• 这个痛点真的重要吗?
• 我的分析有bias吗?
```
